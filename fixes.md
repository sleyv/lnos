# Отчет об исправлении архитектурных, системных и криптографических уязвимостей в LNOS 🛠️

В ходе глубокого рефакторинга и доработки проекта LNOS (Local Network Overlay System) были устранены критические уязвимости безопасности, решены проблемы неопределенного поведения компилятора (UB), исправлены многопоточные гонки данных (Data Races), а также добавлена современная функциональность кроссплатформенного сетевого взаимодействия, системного разрешения имен (glibc NSS) и автоматического тестирования.

Ниже представлен детальный разбор всех выполненных исправлений.

---

## 1. Память (C++): Устранение Undefined Behavior в структурах пакетов

### Проблема
Изначально в `liblnos/include/lnos/protocol.h` структура `Packet` содержала объединение (`union`):
```cpp
union PacketAs {
    PacketAnnounce announce;
    ~PacketAs() {}
};
```
Поскольку тип `PacketAnnounce` содержит нетривиальные члены с динамическим выделением памяти (`std::string name` и `std::vector<Service> services`), использование стандартного `union` без явного ручного вызова конструкторов копирования, перемещения и деструкторов приводило к **Undefined Behavior (UB)**, утечкам памяти и порче кучи при удалении объектов `Packet`.

### Решение
Мы полностью избавились от `union PacketAs`, заменив его на прямое хранение поля типа `PacketAnnounce` внутри структуры `Packet`:
```cpp
struct Packet {
    std::string version;
    PacketType type;
    PacketAnnounce announce; // Прямое хранение без небезопасных union
    std::array<std::uint8_t, PUBLIC_KEY_SIZE> publicKey;
    std::array<std::uint8_t, SIGNATURE_SIZE> signature;
    ...
};
```
Все обращения к полям анонса пакета были упрощены: вместо `p.as.announce.name` теперь используется прямое и безопасное `p.announce.name`. Это полностью исключает UB, упрощает читаемость кода и гарантирует автоматический вызов деструкторов стандартных контейнеров.

## 2. Гонки данных (Data Races) в реестре узлов демона `lnosd`

### Проблема
Демон `lnosd` запускает независимые потоки, которые параллельно читали и писали в глобальный ассоциативный контейнер `std::unordered_map<std::string, Node> nodes`.
В C++ параллельная запись и чтение стандартных контейнеров без синхронизации приводит к **Data Race**, порче внутренней структуры контейнера и неминуемому аварийному завершению приложения (Segmentation Fault) под нагрузкой.

### Решение
Мы заменили простой `std::mutex` на разделяемый мьютекс `std::shared_mutex nodesMutex` из C++17/20, который реализует паттерн блокировки "Один писатель, много читателей" (Reader-Writer Lock):
- **Писатели** (`receiver_ipv4`, `receiver_ipv6`, `cleanup`) используют эксклюзивную блокировку `std::unique_lock<std::shared_mutex>`:
  ```cpp
  std::unique_lock<std::shared_mutex> lock(nodesMutex);
  nodes[announce.name] = { ... };
  ```
- **Читатели** (`printer`, `query_server`) используют разделяемую блокировку `std::shared_lock<std::shared_mutex>`:
  ```cpp
  std::shared_lock<std::shared_mutex> lock(nodesMutex);
  for (const auto& n : nodes) { ... }
  ```
Это позволило полностью защитить контейнер `nodes` от повреждения памяти, сохранив высокую производительность за счет параллельного чтения.

## 3. Криптография и Безопасность: Внедрение проверки подписи пакетов

### Проблема
В первоначальном коде `lnosd` функция `receiver` декодировала входящие пакеты и сразу же вносила узел в таблицу `nodes`. Никакой проверки подписи Ed25519 не производилось! Любой злоумышленник в локальной сети мог отправить пакет с подделанным именем, и демон бы беспрепятственно перезаписал IP-адрес легитимного узла в реестре.

### Решение
Мы реализовали полноценную валидацию подписи входящих пакетов:
1. В `liblnos/src/crypto.cpp` добавлена функция `verifyPacket`:
   ```cpp
   bool verifyPacket(const Packet& packet) {
       Blob data = encode(packet, false); // Кодируем пакет без самой сигнатуры
       return crypto_sign_verify_detached(
           packet.signature.data(),
           data.data(),
           data.size(),
           packet.publicKey.data()
       ) == 0;
   }
   ```
2. В потоках-приемниках демона перед обновлением реестра вызывается проверка подписи:
   ```cpp
   if (!lnos::verifyPacket(p)) {
       std::cerr << "[error] packet signature verification failed\n";
       continue; // Невалидные пакеты отбрасываются
   }
   ```
Теперь подделать пакеты невозможно — злоумышленник не сможет заявить права на чужое имя без знания соответствующего приватного ключа.

## 4. Ошибки Выравнивания памяти (Strict Aliasing & Unaligned Memory Access UB)

### Проблема
В функциях чтения из бинарного буфера `encodedPacketConsumeImpl` выполнялось приведение сырых байтов `char*` к указателям типов `uint64_t*` и `uint16_t*`:
```cpp
data = *(uint64_t *) packet.data; // Ужасное UB!
```
Это приводило к двум тяжелым последствиям:
1. **Нарушение Strict Aliasing**: компилятор мог некорректно оптимизировать код, считая, что указатели разных типов не могут ссылаться на одну и ту же область памяти.
2. **Unaligned Access (Невыровненный доступ)**: На процессорах с архитектурой ARM, MIPS или s390x чтение 64-битного числа по адресу, не кратному 8 байтам, вызывает аппаратное исключение шины (Bus Error) и мгновенный краш.

### Решение
Мы переписали эти функции с использованием абсолютно безопасной и оптимизированной компилятором функции `std::memcpy`:
```cpp
inline bool encodedPacketConsumeImpl(EncodedPacket& packet, uint64_t& data) {
    if (packet.len < sizeof(uint64_t))
        return false;
    uint64_t raw;
    std::memcpy(&raw, packet.data, sizeof(raw)); // Идеально выровненное копирование
    data = be64toh(raw);
    packet.data += sizeof(uint64_t);
    packet.len -= sizeof(uint64_t);
    return true;
}
```

## 5. Независимость от Архитектуры: Поддержка Порядка Байтов (Endianness)

### Проблема
Изначально сериализация чисел выполнялась в порядке байтов хоста (Host Byte Order). Если одна машина в сети работает на процессоре Little Endian (например, x86_64), а другая — на Big Endian (например, некоторые роутеры на MIPS/PowerPC), они не могли понять пакеты друг друга, так как байты чисел приходили в обратном порядке.

### Решение
Мы привели передачу всех многобайтовых полей к стандартному сетевому порядку байтов (Big Endian):
- При кодировании (`blobPush`):
  ```cpp
  inline void blobPush(Blob& blob, uint64_t data) {
      uint64_t be_data = htobe64(data); // Преобразование в Big Endian
      const uint8_t *ptr = reinterpret_cast<const uint8_t*>(&be_data);
      for (uint64_t i = 0; i < sizeof(be_data); ++i)
          blob.push_back(ptr[i]);
  }
  ```
- При декодировании (`encodedPacketConsumeImpl`):
  ```cpp
  data = be64toh(raw); // Восстановление порядка байтов хоста
  ```

## 6. Сеть: Автоматическое Определение Интерфейсов и IPv4/IPv6 Dual-Stack

### Проблема
Раньше демон работал только на жестко заданном IPv4-адресе, а переменная `myIp` оставалась пустой, из-за чего сокеты не могли связаться с реальными сетевыми интерфейсами. Также отсутствовала поддержка современного протокола IPv6.

### Решение
Мы разработали динамическую систему сетевого взаимодействия:
1. **Автоопределение с `getifaddrs()`**: Демон опрашивает сетевые адаптеры ОС, выбирает первый активный физический (не loopback) интерфейс и извлекает его IPv4 и IPv6 адреса.
2. **Параллельный Dual-Stack**: Демон теперь запускает независимые потоки для IPv4 и IPv6:
   - `sender_ipv4` и `receiver_ipv4` работают с multicast-группой `239.255.42.99`.
   - `sender_ipv6` и `receiver_ipv6` работают с link-local multicast-группой `ff02::4299`.
Это гарантирует, что система бесперебойно работает в любых современных сетях с поддержкой IPv6.

## 7. Разрешение Конфликтов Связывания Портов (IPV6_V6ONLY)

### Проблема
При запуске параллельных IPv4 и IPv6 приемников на одном и том же порту `4545` возникал конфликт: по умолчанию в Linux сокет IPv6, связанный с `in6addr_any` (wildcard), также прослушивает входящие IPv4 соединения. Это приводило к тому, что один из сокетов выбрасывал исключение `EADDRINUSE` (Address already in use), и приемник падал.

### Решение
Мы добавили вызов `setsockopt` с флагом `IPV6_V6ONLY` на IPv6 сокете, гарантируя полное разделение протоколов и их параллельное функционирование без взаимных конфликтов:
```cpp
int v6only = 1;
if (setsockopt(sock, IPPROTO_IPV6, IPV6_V6ONLY, &v6only, sizeof(v6only)) < 0) {
    stopAfterSystemError("receiver_ipv6 IPV6_V6ONLY");
    close(sock);
    return;
}
```

## 8. Интеграция с Реальной ОС: Локальный Сервер Запросов и Модуль NSS (Name Service Switch)

### Проблема
Без интеграции с системным резолвером ОС, утилиты системы (`ssh`, `ping`, `curl`) не могут использовать имена LNOS (например, `pc.main.gervaty`), что делало проект нежизнеспособным вне рамок тестовой консоли.

### Решение
Мы создали полноценную цепочку разрешения имен на уровне операционной системы:
1. **UNIX Domain Socket Query Server**:
   В `lnosd` запущен локальный сервер запросов, слушающий на `/tmp/lnosd.sock`. Он принимает текстовые запросы вида `node.gervaty\n`.
2. **Distributed Active Query-Response Protocol**:
   Если запрашиваемое имя найдено в локальном кэше — IP возвращается мгновенно. Если имени нет (cache miss), `lnosd` активно рассылает в multicast бинарный пакет типа `PacketType::Query`. Получатель с таким именем мгновенно отвечает прямым unicast пакетом `PacketType::Response`. Сервер засыпает на 400мс, дожидается ответа и возвращает полученный IP клиенту.
3. **NSS-модуль (`libnss_lnos.so.2`)**:
   Мы реализовали нативный glibc NSS модуль (`liblnos/src/nss_lnos.cpp`), экспортирующий функции `_nss_lnos_gethostbyname2_r` и `_nss_lnos_gethostbyname_r`. Он подключается к локальному `/tmp/lnosd.sock`, отправляет запрос резолвинга и преобразует строковый IP в структуру `struct hostent` операционной системы.

## 9. Спецификация XDG и Работа Без Root-прав

### Проблема
Все пути к файлам конфигурации были жестко зашиты в коде, требуя обязательного запуска утилит от имени суперпользователя (`root`), что критически неудобно для пользователей и небезопасно для системы.

### Решение
Мы реализовали функцию `getConfigDir()`, которая автоматически определяет домашнюю директорию и права текущего пользователя, используя `$XDG_CONFIG_HOME/lnos` или `~/.config/lnos`. Обычные пользователи теперь могут генерировать ключи и использовать `lnosctl config/set/get`, не повышая привилегий.

## 10. Безопасность Потоков: Исключение Падений и Graceful Shutdown

### Проблема
Изначально ошибки сети или отсутствие файлов ключей приводили к необрабатываемым исключениям и падению отдельных потоков, тогда как другие продолжали выполняться, переводя приложение в состояние "зомби".

### Решение
Все критические точки упакованы в безопасные блоки `try-catch`. При возникновении любой критической ошибки в сокетах или файловой системе вызывается `stopWithError()`, атомарно выставляющая глобальный флаг `running = false` и закрывающая ресурсы. Это гарантирует одновременное и чистое завершение всех потоков демона.
